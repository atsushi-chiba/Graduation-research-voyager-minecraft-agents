# 研究の位置づけ

## 何を明らかにしようとしているか

大規模言語モデル (LLM) が長時間・自律的に「統治」の意思決定を続けたとき、どういうふるまいを見せるか。人間の指示なしに、限られた情報と行動オプションのなかで、LLM は都市をうまく回せるのか。回せないなら、どこで詰まり、どんな失敗をするのか。

本研究では Minecraft の街づくり mod である **MineColonies** を実験環境として選び、LLM に「市長」の役割を与え、コロニー (町) の運営判断を全面的に任せます。市民 NPC は既存の MineColonies の仕組みで自律的に建設・生産・戦闘を行うため、LLM は「都市計画者」に徹し、細かな身体労働を書かせる必要がありません。

## 最終目標

### ① 2コロニー間の戦争

独立に動く 2 つの LLM 市長にそれぞれ別のコロニーを与え、有限な資源 (領土・木材・鉄鉱石・食料) を巡って自然に競合が発生するかを観察します。外交メッセージ、資源略奪、軍備拡張、そして開戦・停戦 — 一連の政治的行動が、単に「戦え」と命令することなく、状況圧力だけで LLM から立ち上がるかを検証します。

### ② 狭小空間での効率的発展

コロニーの領土を意図的に狭く制限した条件下で、市長がどのような優先順位で建物を配置するかを追います。無限に広がる土地では暴力的に建て増しできてしまうため、リソース最適化やレイアウト工夫といった "都市計画" の質は評価しにくい。制約下でこそ、LLM の設計判断のクセや盲点が観察できると期待しています。

### ③ 市民 NPC の SNS 発言

MineColonies の市民は幸福度・空腹度・健康状態などの内部状態を持ちます。それらを言語化した "つぶやき" を、OASIS 等の SNS プラットフォームに流し込むことで、コロニー全体を "住民の声を持った社会" として観察できるようにします。市長の政策と市民の反応を、政治家・有権者関係として可視化することが狙いです。

## なぜ Minecraft / MineColonies か

- **既存のゲームロジックが "労働" を代替してくれる** — 市民 NPC は伐採・農耕・建設を勝手にこなす。LLM は行動の詳細ではなく判断だけを担えばよく、実験の焦点が明確になる
- **状態を数値化しやすい** — 建物レベル、市民人口、幸福度、資源在庫、研究進捗などが全て内部数値として観測可能
- **時間の圧縮が可能** — サーバーの tick 速度を制御することで、現実の数時間で数ゲーム日を観測できる
- **エージェント間の非対称性を作りやすい** — 領土サイズ、初期資源、地形、市長のプロンプトなどを容易に変えて A/B 比較が可能

一方で、`mineflayer` (LLM が Minecraft を "操作する" 標準的な OSS ライブラリ) は Forge サーバーに接続できないという制約があります。このため本研究では独自の Forge mod ("VoyagerBridge") をサーバー内部に組み込み、HTTP API 経由で LLM が MineColonies の各機能を呼び出す方式を採用しました。詳細は [システム構成](system.md) を参照してください。

## 実験のフェーズ

現在は目標 ②③ の前段として、**通常ワールド (起伏あり) での自給自足コロニー** を安定稼働させる段階です。以前はスーパーフラット (完全な平地・鉱石なし) で動かしていましたが、「miner が死に職になる」「supply_bot が全資源をチートで供給し続ける」といった歪みが観察できたため、通常ワールドへ移行しました。地形対応の建物配置アルゴリズム (heightmap サンプル + 平坦さゲート) と、時間経過とともにチート供給を減衰させる仕組みを新たに実装しています。

次のフェーズで目標 ① (2 コロニー戦争) と目標 ③ (SNS 連携) に着手する予定です。
